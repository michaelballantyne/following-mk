# Fresh tails, and why not-in-envo/d is disabled

In `restricted-interp-following.scm`, every call to `not-in-envo/d` is
commented out (`#;`) and its definition is commented out too. The TODO
comment above the disabled definition reads:

> is it good or bad that this stalls when it gets to ambiguity, despite
> being used in guard?

The answer turned out to be: *bad*. Here's why.

## What "fresh tail" means

A fresh tail is an environment list whose spine is concrete for a few cells
but ends in an unbound logic variable rather than `'()`:

```
((k1 . v1) (k2 . v2) ... . X)
                             ↑
                        fresh logic var —
                        could still become '()
                        or another cons cell
```

In ordinary miniKanren this is normal: env gets built incrementally by
unification, so at any given moment the rest of env past the portion that's
been committed to may still be a logic variable the search hasn't pinned
down yet.

## Why that breaks not-in-envo/d in a guard

`not-in-envo/d` is a recursive `conde/d` over the env spine:

```scheme
(conde/d
 ([] [(==/d empty-env/d env)] [])
 ([y b rest]
  [(==/d `((,y . ,b) . ,rest) env) (=/=/d y x)]
  [(not-in-envo/d x rest)]))
```

When the walk reaches a fresh-tail variable `X`, *both* clauses are
independently consistent with it:

- clause 1's guard would bind `X` to `'()` — singleton success
- clause 2's guard would bind `X` to `(cons (y . b) rest)` with
  `(=/=/d y 'cons)` — singleton success, because `y` is itself fresh and
  the disequality trivially holds

Two clauses both commit singleton → `conde/d-runtime` classifies the whole
call as multi-commit → returns `'nondet`.

That `'nondet` propagates up into the *outer* `eval-expo/d` clause's guard
— typically one of `quote`, `cons`, `letrec`, `match`, `if` — which was
otherwise happily about to commit on a syntactic-form check. The outer
guard is now classified as ambiguous, so the whole interpreter stalls or
starts returning suspension thunks on what should be an obviously
committable step. Divergence follows, because the interpreter never makes
real progress.

## The workaround

Commit `d845732` ("some conde/d interp tweaks to try and avoid divergence…")
disabled all five `not-in-envo/d` calls (quote, cons, letrec, match, if)
and the definition itself. The same commit also replaced the
function-application clause's guard `(lookupo/d rator env cv ct)` (another
env-walk that suffers the same problem) with

```scheme
(symbolo/d rator) (absento/d rator '(quote cons letrec match if))
```

— a guard that inspects `rator` alone and asks zero questions about env,
so it can always commit promptly.

## What this costs

The interp is now *unsound* on programs that rebind `quote`, `cons`,
`letrec`, `match`, or `if` as local variables (shadowing keywords). That's
an acceptable corner for the synthesis benchmarks because none of them do
that. But it's a real corner and worth remembering if the benchmark set
ever grows.

## The deeper lesson

A `conde/d` guard is only useful if it can reach a commit decision from
*locally available information*. Any guard that recurses over structure
containing fresh tails risks turning into a nondet-emitter, which is worse
than useless because the nondet poisons whatever outer guard contains it.
Design guards to depend only on information the main search has already
determined by the time the guard runs.
