# Counters: what they measure and how to read them

`following.scm` keeps a handful of counters that `print-counters!` dumps
at the end of every `run`.  This note is the reference for what each one
means and how to use them when judging whether the follower is earning
its keep.

## The counters

- **`*==-counter*`** — calls to the top-level `==` (main-search
  unification).  The plain `==` in `following.scm` is a wrapper around
  `==-base` that just ticks this counter before dispatching.  Note that
  `fresh`, `run`, and the `conde` / `mk.scm` internals call the underlying
  unifier directly and don't tick this, so it undercounts slightly; it's
  still a useful rough signal of how much main-search unification work
  happened.

- **`*==/d-counter*`** — unifications performed by `==/d` inside a
  follower goal.  This is a wrapper around `==-base` that ticks before
  dispatching, installed specifically so the instrumentation can tell
  "work done inside the follower" apart from "work done in the main
  search."  Big `==/d-counter` = the follower is doing a lot of
  interp-style evaluation.

- **`*fail-counter*`** — times the follower was triggered and its goal
  evaluated to `#f` (empty stream).  Each one kills the outer search
  branch the trigger fired on.  **This is the follower's "refutation
  pruning" signal:** every increment is a branch the main search didn't
  have to explore.

- **`*singleton-succeed-counter*`** — times the follower was triggered
  and its goal evaluated to a single state with no remainder.  The
  follower is fully discharged on this path: its F slot gets cleared
  (`state-with-F c^ #f`) and no re-triggering will happen here again.

- **`*non-singleton-succeed-counter*`** — times the follower was
  triggered and its goal evaluated to `(state . resume-thunk)`.  The
  committed state `c^` flows out; the resume thunk is stashed back in F
  for later re-firing.  This is the *suspend* path.

- **`*externally-productive-trigger-counter*`** — times a trigger's
  committed state `c^` changed `(walk* term S)` for the follower's
  outer-visible `term`, as measured against the state walk of `term`
  before the trigger ran.  **This is the follower's "propagation"
  signal:** every increment is a trigger that taught the outer search
  at least one new fact about a variable reachable from `term`.

- **`*externally-unproductive-trigger-counter*`** — the complement: a
  trigger that didn't change the walked term.  The trigger may still
  have done useful *internal* work (setting things up so a later trigger
  can commit), but as of that trigger it produced nothing the outer
  search can see.

- **`*unsound-fail-depth-cutoff-counter*`** — hit the unsound depth
  cutoff and failed outright.  Should be zero unless
  `*unsound-fail-depth*` has been dropped from `+inf.0` for debugging.
  A non-zero value here means the follower returned spurious failures;
  the resulting run is no longer sound.

- **`*suspend-depth-cutoff-counter*`** — hit the (sound) depth cutoff
  and suspended with a resume thunk.  Non-zero here means the follower
  was cut off mid-evaluation; it's sound, but depth-limited in how far
  it chased.

- **`*main-unsound-depth-cutoff-counter*`** — number of times the
  patched `conde` in `mk.scm` saw `state-D` exceed `*main-unsound-depth*`
  and failed the branch outright.  Parallel to
  `*unsound-fail-depth-cutoff-counter*` but on the main search rather
  than the follower's internal search.  Non-zero means the run is no
  longer sound; zero by default since `*main-unsound-depth*` defaults
  to `+inf.0`.

- **`*user-counter*`** — manual instrumentation counter.  Ticked by the
  `(user-count)` goal, available for ad-hoc probing in a test.

## Two modes of follower value

There are two ways a follower can help the main search:

1. **Refutation** — the follower runs and decides the current branch is
   infeasible, killing it.  Signal: `*fail-counter*`.  Each increment
   prunes a subtree the main search would otherwise have explored.
2. **Propagation** — the follower runs, commits one or more new
   bindings to outer-visible variables (`c^` is richer than `st` on the
   walk of `term`), and those bindings show up in the outer search.
   Signal: `*externally-productive-trigger-counter*`.  Each increment
   teaches the outer search something it would have had to derive on
   its own.

These are genuinely different mechanisms and both are "the follower
earning its keep," but they're not the same kind of work:

- Propagation *is not* pruning.  It's forward inference: the follower
  committed a fact that narrows future search.  It doesn't kill any
  branch directly; it just gives the outer search more info to work
  with.
- Pruning *is not* propagation.  It's backward inference: the follower
  proved the current branch inconsistent and cut it.  It doesn't commit
  any new bindings — the state is just gone.

Both ultimately reduce total search work, but they do so in different
ways and the counters distinguish them.

## Reading the results

The sanity check: `externally-unproductive` — `fail` = "triggers that
neither refuted the branch nor taught the outer search anything new."
If this number is large relative to total triggers, the follower is
being asked to re-prove things and not making forward progress per
trigger.  Each such trigger is pure overhead.

That's not necessarily wrong — a follower in "suspend" mode might be
making internal progress across triggers and only commit externally
after several have piled up.  But if the whole run ends with
`externally-productive = 0` and `fail = 0`, the follower contributed
*nothing* to the outer search on this benchmark.  It did work, but
nobody saw any of it.

The `synthesis-small.scm` micro-benchmark is an example of this: on
that task, `externally-productive` is 0 and `fail` is 124, so the
follower is pruning but not propagating.  The baseline meanwhile solves
the task in well under a millisecond because the main-search `evalo`
refutes the same 124 (or similar) branches cheaply — so the follower's
pruning work is redundant and its ~70K internal `==/d` calls are
overhead.

## Caveats

- **Unproductive ≠ wasted in principle.**  A trigger that didn't change
  the walked term may have committed bindings to *internal* follower
  fresh vars that a later trigger will use to derive an outer-visible
  fact.  The counters can't distinguish "useless step" from "useful
  setup for a future step" on a per-trigger basis.  Look at the final
  totals, not individual triggers.
- **Large overhead is expected in this repo for now.**  The question is
  not "is the follower cheap" — it isn't — but "can it produce
  significant pruning (`fail`) or propagation (`externally-productive`)
  at all?"  If yes on some benchmark, the next question is how to do
  the same work more cheaply.  If no, the mechanism isn't pulling its
  weight at all.
- **`==-counter` and `==/d-counter` are not directly comparable.**
  They measure different call sites: the former covers main-search
  unification via the overridden `==`, the latter covers follower
  unification via `==/d`.  They can have very different ratios on the
  same task depending on where the work is happening.
