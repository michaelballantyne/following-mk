# Design notes

Material here is the stuff a reader of the README might want next:
the mk.scm patch detail, the soundness story behind extension commitment,
the relationship to the classical underconstraint form, and the naming
history. The README covers *what* the thing is and how to use it; this
note covers *why* it's shaped the way it is.

## Alternate framings for the same idea

The README names the Andorra Principle. The same mechanism shows up under
other labels in other communities:

- **Unit propagation** (SAT/SMT): apply every implication that's forced
  by the current partial assignment; defer anything that still has real
  choice.
- **And-parallelism / coroutining** (Prolog): run conjuncts that don't
  yet have an ambiguous goal ahead of the scheduler's linear march
  through them.
- **Determinacy-directed evaluation**: interpret the program
  relationally, but only take steps where the interpreter has enough
  info to commit.

"Determinacy-directed" is the phrase used in this repo's identifiers and
prose; "Andorra" is the standard external reference.

## The mk.scm patches

Only three spots in `mk/mk.scm` are touched. The goal is to keep the fork
against upstream small and auditable.

1. **State shape.** `state` becomes a 3-slot list carrying an `F`
   (follower) slot alongside `S` (substitution) and `C` (constraint
   store). New accessors `state-F` and `state-with-F` are exported.
   `empty-state` carries `#f` in F.

2. **`==`.** Threads `(state-F st)` through the state it constructs on
   unification success, so the follower cell survives a unification step.
   Without this, unification would drop the stashed resume thunk.

3. **`conde`.** Binds against `((trigger-followers) st)` before each
   branch. This is the hook that lets a stored follower re-fire at each
   choice point as the search learns more. `trigger-followers` is a free
   identifier at macro-definition time; it's resolved when `following.scm`
   is loaded later.

Everything else in `following.scm` is pure add-on: no further edits to
`mk/`. `trigger-followers` is also called once at the end of `run` in
`following.scm`, just before reification, as a final sanity check.

## Follower trigger points, in one place

- **Immediate.** `(follower name term goal)` runs `goal` once right away
  against the entry state. Three outcomes: fail (outer search fails),
  singleton success (drop the follower, commit and continue), suspended
  (stash `(resume-thunk . term)` in `state-F` and continue with the
  extensions made so far).
- **At each `conde` choice point.** The patched `conde` calls
  `(trigger-followers)` on the state before branching, so a stashed
  follower re-fires on each branch's state as new info appears.
- **At end of run.** `run` calls `(trigger-followers)` once before
  reification so any remaining follower work is evaluated before the
  answer is shown.

## Extensions are committed: the key divergence from underconstraints

**The non-obvious design point.** When a `conde/d` wrapped in a follower
commits a clause, the substitution and constraint-store extensions made
by the committed body *do* flow out to the outer search state. Only the
`F` slot is updated on the entry state; everything else passes through
from the result state, not the entry state. The relevant code in
`run-and-set-follower`:

```scheme
(case-inf/d $
  (() #f)
  ((c^) (state-with-F c^ #f))
  ((c^ f^) (state-with-F c^ (cons f^ t))))
```

`c^` is the state *resulting from* running the follower's goal. There is
no rollback to the entry state — whatever the follower committed, the
outer search sees.

This is a meaningful departure from classical one-shot/general
underconstraints, where the whole point is that the check is a
*necessary-but-not-sufficient* predicate with no effect on the
constraint store. Underconstraints in that tradition can be evaluated
independently of constraint interaction precisely because they never
write back. What's in this repo is closer to a `onceo`-wrapped
interpreter step: extensions that `conde/d` is *forced* to make (because
exactly one clause committed given current info) flow out to the outer
store.

### Soundness argument

If `conde/d` only commits a clause when that clause is uniquely
determined by the current state, then any extensions the committed body
makes are extensions a sound evaluation of the same relation on the same
state would also make. Committing them early to the outer store is sound
in exactly the same sense the committed step is sound: we are executing
a step the main search was going to execute anyway, just earlier.

The load-bearing piece of this argument is that `evaluate-guard`
correctly reports `'nondet` whenever there is real ambiguity. A bug
there — classifying an ambiguous guard as singleton — would leak
unforced extensions to the outer store. That's the correctness
invariant to be careful about when touching guard evaluation.

## `conde/d` evaluation rule, spelled out

For each clause, run its guards against the current state and classify
the resulting stream:

- **empty** → clause ruled out
- **singleton** → clause is a candidate to commit
- **multi-answer** → `'nondet` (ambiguous)

Then decide:

- exactly one candidate, no ambiguity → **commit**: run that clause's
  body against the (guard-extended) state
- ≥2 candidates, or any `'nondet` → call `(nondeterministic)`, which
  returns `(cons st g-thunk)` — hand back the *entry* state paired with
  a thunk that re-runs the whole `conde/d` later, when more info is
  known
- zero candidates → **fail**

The entry state (not the guard-extended one) is what gets handed back in
the nondet case. That's what makes stalling safe: nothing the stalled
clauses wrote is visible to the outer search.

### Determinacy is liberal: guards may extend the store

"Singleton" means the guard produces one result state, not that the guard
left the store untouched. A guard like `(==/d x 1) (=/=/d y 2)` still
singleton-succeeds, and those extensions travel with the commit.

This is sound because `conde/d-runtime` evaluates every clause's guard
against the common entry state `st` — not against a previously-committed
clause's extended state. If exactly one clause's guard singleton-succeeds
on `st`, that's the unique step any sound evaluation would take, and its
extensions are the extensions that step would make.

## Naming history

The form used to be called `condg` (for "guarded conde"), its primitives
were suffixed `/g`, and the installer was `underconstraino` (inherited
from Will Byrd's underconstraints exploration). Those names were
renamed:

- "underconstraint" suggested the classical
  necessary-but-not-sufficient semantics, which isn't what this
  mechanism actually does — see the extension-commitment section above.
- `condg` described only the structure (guarded cond), but `==/d`,
  `fresh/d`, etc. don't have guards at all. The real shared property is
  that they participate in a determinacy-directed evaluation mode. `/d`
  names that directly.
- "follower" captures the ongoing registration aspect of the installer:
  the wrapped goal *follows* the main search and re-fires, rather than
  running once and done.

The current names aren't set in stone, but they're at least honest about
what the mechanism does.
