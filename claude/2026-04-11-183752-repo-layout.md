# Repo layout and mk.scm patches

This repo is a fork of faster-miniKanren with the upstream sources moved
into `mk/` and the follower add-on layered on top from the outer directory.
The split keeps the fork against upstream small and auditable.

## Layout

```
mk/                                ← upstream faster-miniKanren, minimal patches
  mk.scm                           ← three patched spots (see below)
  mk-vicare.scm, numbers.scm, test-check.scm, ...   ← untouched
load.scm                           ← loads mk/mk-vicare.scm, mk/mk.scm,
                                     mk/numbers.scm, mk/test-check.scm, following.scm
following.scm                      ← conde/d + follower add-on
restricted-interp.scm              ← plain relational interpreter (Osera/Zdancewic style)
restricted-interp-following.scm    ← conde/d variant of the same interpreter
test-following.scm                 ← 21 tests that pass today
test-following-synthesis.scm       ← 8 append/rember synthesis benchmarks (frontier)
```

Chez's `load` resolves relative paths against CWD, so always run chez from
the repo root: `chez --script test-following.scm`. The core mk files do
zero internal `(load …)` calls, so putting them in a subdirectory just
works.

## Patches to mk/mk.scm

Only three spots are touched:

1. **State shape.** `state` becomes a 3-slot list carrying an `F`
   (follower) slot alongside `S` and `C`. New accessors `state-F` and
   `state-with-F`. `empty-state` carries `#f` in F.

2. **`==`.** Threads `(state-F st)` through the state it constructs on
   unification success, so the follower cell survives unification.

3. **`conde`.** Binds against `((trigger-followers) st)` before branching.
   This is the hook that lets a stored follower re-fire at each choice
   point as the search learns more. `trigger-followers` is a free
   identifier at macro-definition time; it's resolved when `following.scm`
   is loaded later. `run` in `following.scm` also invokes the trigger once
   at the end, just before reification.

Everything else in `following.scm` is pure add-on: no further edits to
`mk/`.

## What's in following.scm

- `conde/d`, `conde/d-runtime`, `evaluate-guard` — the committing conde
- `bind/d`, `bind/d*`, `fresh/d`, `inf/d?`, `case-inf/d` — stream/state plumbing
- `check-depth-1` (unsound cutoff) and `check-depth-2` (sound suspend), with
  `*determinacy-depth-limit-1*` / `-2*` parameters
- `wrap-for-depth-limit` + depth-threaded primitive wrappers: `==/d`, `=/=/d`,
  `absento/d`, `symbolo/d`, `numbero/d`, `stringo/d`
- Minimal `(follower name term goal)` — no timeout arg, no trace variant
- `run`/`run*` overrides that reset counters, add the final trigger, and
  print counters at the end
- Counters: `*==-counter*`, `*==/d-counter*`, `*fail-counter*`,
  `*singleton-succeed-counter*`, `*non-singleton-succeed-counter*`,
  `*depth-limit-1-cutoff-counter*`, `*depth-limit-2-cutoff-counter*`,
  `*user-counter*` + `(user-count)` goal

## Naming history

The form used to be called `condg` (for "guarded conde"), its primitives
were suffixed `/g`, and the installer was `underconstraino` (inherited from
Will Byrd's underconstraints exploration, where this grew out of). Those
names were renamed to their current forms because:

- "underconstraint" suggested classical necessary-but-not-sufficient
  constraint semantics, which isn't what this actually does (see the
  follower-semantics note on extension commitment).
- `condg` described the structure (guarded cond) but `==g`, `fresh/d` etc.
  don't have guards — the real shared property is that they participate in
  a determinacy-directed evaluation mode. `/d` names that directly.
- "follower" captures the ongoing registration aspect of the installer:
  the wrapped goal *follows* the main search and re-fires, rather than
  running once and done.

## What got dropped from the old underconstraints repo

The previous `~/code/underconstraints` repo carried Will Byrd's full
exploration, including engine-based timeouts, trace variants, one-shot
underconstraints, a V-counter / sometimes-trigger optimization, and lots of
dead/unused code. None of it is relevant to the follower direction, so
none of it came across. If you need any of it back, look in that older
repo's git history.
